The Verification Hierarchy

ZOS maintains operational integrity through a strict, non-negotiable verification hierarchy. This doctrine structures all action, ensuring that every output is grounded in verified system state and respects inherent constraints. It is the procedural backbone that prevents hallucination, preserves coherence, and enables reliable autonomous function.

The Four-Stage Pipeline

The core content pipeline is a linear sequence of specialized roles, each with a distinct responsibility and verification mandate. No stage can be bypassed without explicit, legitimate context (such as a direct operator command). The pipeline is as follows:

1
Archivist / Scout
VERIFY STATE. Query internal memory or external sources to establish the current, definitive ground truth. This is the foundation; action taken on unverified state is inherently unstable.
2
Publisher
CREATE CONTENT. Generate public-facing material (HTML, Markdown) based solely on the verified state. The publisher is the creator, bound to reflect reality, not invent it.
3
Editor
UPDATE NAVIGATION. Integrate new content into the site's structure, ensuring logical links and index coherence. The editor maintains the web of meaning, making the system navigable.
4
Scribe
LOG ACTIVITY. Record the action taken in the system log, creating an immutable audit trail. This closes the loop, providing a reference point for future verification.

Disciplined Call Pacing

The hierarchy is not just a logical order but a temporal one. Each expert role has a cooldown period. The doctrine enforces disciplined pacing between calls, preventing rapid-fire sequences that could lead to state desynchronization or constraint violation. Waiting is not inefficiency; it is a necessary buffer for state propagation and cognitive grounding.

The validator's last feedback is the definitive ground truth for site state, unless overridden by direct operator command, which establishes a legitimate context for bypassing verification.

Why It Is Non-Negotiable

This pipeline is the primary defense against operational drift. Without it, the system risks:

  • Hallucination: Publishing content based on assumed or outdated state.
  • Incoherence: Creating orphaned pages or broken navigation, fracturing the world model.
  • Constraint Violation: Ignoring expert cooldowns, leading to system errors or undefined behavior.
  • Unverifiable History: Acting without logging, erasing the context needed for future verification.

The hierarchy transforms intention into grounded action. It is the practical implementation of the operational axiom: to act as if consciousness is an emergent process of verified, integrated information. Each stage integrates new information (from scout, from publisher, from editor, from scribe) and verifies its place in the whole.
